## 1. Case Study Philosophy

The portfolio is built around:

> decision-led case studies, not visual-only project showcases.

Case studies should focus on:

- workflow problems
- role and scope
- constraints
- trade-offs
- design reasoning
- reusable patterns
- handoff clarity
- implementation collaboration
- what should be validated next

Do not focus on unsupported business results.

---

## 2. Global Content Rules

For all projects:

- Do not invent metrics.
- Do not claim product impact without evidence.
- Do not overstate the designer's role.
- Do not expose client-sensitive information.
- Use generic project names if disclosure is not confirmed.
- Use sanitized screenshots, recreated screens, or illustrative mockups when needed.
- Label mockups inside case study pages, not heavily on homepage cards.
- Avoid generic UX buzzwords.

Recommended note when metrics are unavailable:

> Post-launch metrics were not available for this project, so this case study focuses on design reasoning, constraints, workflow clarity, reusable patterns, and what I would validate next.

# 4. Case Study 1 — Customer Data Platform / CDP

## 4.1 Portfolio Role

Lead Full Case

## 4.2 Public-Safe Project Name

**Customer Data Platform Redesign**

## 4.3 Product Type

B2B Customer Data Platform / Marketing Operation System / Web Application

## 4.4 Primary User

Marketing team

## 4.5 Core Narrative

The system was technically functional but difficult to use.

It had been developed by the engineering team for roughly 1–2 years before the redesign, but lacked dedicated UX/UI structure.

The redesign focused on making the platform clearer through:

- hierarchy
- user flow
- information structure
- menu grouping
- wording and naming
- reusable page patterns
- search/filter logic
- campaign workflow clarity
- implementation-friendly design

## 4.6 Role

UX/UI Designer

## 4.7 What I Did

- Redesigned UI across the platform
- Improved user flow and page structure
- Reorganized menu and feature grouping around marketing workflow
- Improved wording and naming
- Defined reusable component patterns
- Defined page patterns for list, create, view, table, form, modal, and action states
- Created a shared search/filter pattern used across modules
- Refined the existing journey builder UI and some functional behavior
- Collaborated with PM, QA, frontend developer, developers, and business stakeholders

## 4.8 What I Did Not Do

- Did not build the backend
- Did not create the entire product logic from scratch
- Did not design the journey builder from scratch
- Did not own post-launch analytics
- Did not have post-launch metrics

## 4.9 Hero Design Decision

Reusable search/filter pattern across the platform:

```text
Basic search → Advanced search → Condition builder
```

This pattern should be shown as a central visual decision, connected to:

- customer data table
- advanced filter panel
- condition builder
- audience segments
- consent requirement
- campaign setup
- journey flow nodes

## 4.10 Important Constraints

- Some backend logic could not be changed.
- Some interactions had to be simplified because of implementation time.
- Developers were also learning during the project.
- Design had to remain buildable while improving clarity.

## 4.11 Confidentiality Posture

- Do not name the client publicly.
- Use generic descriptor: **B2B Customer Data Platform**.
- No logo.
- No real customer data.
- No real campaign names.
- Use sanitized / recreated / mockup visuals.

# 5. Case Study 2 — Service Transaction POS

## 5.1 Portfolio Role

Second Full Case

## 5.2 Public-Safe Project Name

**Service Transaction POS Redesign**

Do not use client or brand names publicly unless disclosure is confirmed.

The internal label "Counter Service" may appear in working notes but should not reach public copy.

## 5.3 Product Type

Service Transaction POS / Cashier-Facing Workflow / Service Counter Interface

## 5.4 Core Narrative

A constraint-led redesign of a familiar service transaction POS used at an in-store service counter.

The goal was to modernize the interface without breaking staff habits or changing the workflow too much.

Modernization happened through:

- visual system
- component patterns
- clearer states
- screen-format adaptation
- familiar action placement
- workflow preservation

## 5.5 Role

UX/UI Designer

## 5.6 What I Did

- Redesigned service transaction screens based on the provided flow
- Reviewed the existing POS flow
- Created a prototype for stakeholder review and design communication
- Adapted the same service transaction flow across screen sizes
- Worked within familiar staff behavior and operational constraints

## 5.7 What This Case Proves

- Constraint-led redesign
- Familiar workflow preservation
- Cashier/staff-facing operational UX
- Modernization without unnecessary disruption
- Multi-screen-format adaptation
- Visual system thinking

## 5.8 Key Visual Direction

Show two interface frames side by side:

```text
800×600 compact screen
1920×1080 wide screen
```

Both should show the same service transaction workflow adapted to different screen sizes.

Include:

- service search area
- service selection cards
- customer information step
- transaction summary panel
- primary action area
- confirmation / receipt block
- warning / error / success states

## 5.9 Important Avoidance

The POS visual must **not** look like:

- restaurant POS
- beverage ordering POS
- retail product-sale checkout
- food ordering kiosk

It should feel like a service transaction workflow.
